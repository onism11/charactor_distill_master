# 蒸馏大王 — 角色蒸馏 pipeline

把角色原始材料（台词、剧情、人物故事、语音与设定）蒸馏成可运行、可审计的角色 skill 包。

## 公开主线

- `SKILL.md` 是唯一蒸馏入口。
- `workflow-stage-*` 是按阶段读取的内部提示资源，不是独立 skill。
- `workflow-prompts/` 是按需启动执行角色的生产控制面；一次只读对应的一份，不进入最终角色包。
- `workflow-stage-b-distill/workflow-stage-b-distill.md` 是唯一公开的 Distill 主线。
- active 不使用 `answer-agent-prompt.md`；F/G 只在用户明确要求答题测试或横评时启动。

当前公开内容是 **S_微调版**：以 Before-B/S 的生产语义为底本，只保留已经明确批准的轻量增量，包括单入口结构、可选 Theme/Skill Checker、S6 解释权提示，以及原始材料超过 200 KB 时才开放 Large Rawcut Handoff 的 S1 前置标记。

## 设计原则

- **注意力预算零和**：扩张 action 或检查模板会挤压 analysis/persona；先保人物，再加纪律。
- **保留混沌，约束外置**：workflow 只设证据与边界下限，不替 persona 规定文体、长度和结论位置。
- **轮子能跑就不要动**：稳定路径默认不重写；只有可复现失败或高代价失败才启动修改。
- **最小改动**：一个问题只动最靠近根因的一层；能用一行解决，就不新增脚本、产物、gate、session 或 checker。
- **优雅极简**：倾向更少的文件、状态、交接和例外；机制复杂度高于问题时，默认驳回。
- **不要过度防备**：只为已发生或高概率、高损失的风险设防，接受可控误差，不追求预防性的全链加固。

## 修改纪律

- 不借结构整理顺手修改 producer/checker 的产品语义。
- 新能力先在隔离副本验证，再决定是否进入主线。
- 每一处改动都应能直接追溯到具体失败或用户裁决。
- 最终角色包与生产工作流分离；controller prompt、checker 报告和原始证据仓不得误装进角色运行时上下文。
