# Distill Skill — 软版

## 软版指引

本版**不强制数量**。每节给 workdown 结构指引 + 好 / 坏范例对比，允许空缺。凡空必须写"材料不足，未展开"。

**重点：质量优先于覆盖率**。能写好的写满，写不好的诚实留空。

软版与硬版的核心差别：

| 维度 | 硬版 | 软版 |
|---|---|---|
| Layer 0 数量 | ≥ 6 条，必填或标注材料不足 | 看材料决定，能写几条写几条 |
| Layer 2 高频词 | ≥ 6 个带出处 | 看角色，不强制 |
| Layer 4 对象 | 强制 5 类 | 按材料覆盖，常见类不强制 |
| 行动方式格式 | 强制"X：通过 Y 使 Z" | 允许混合格式，能传达"怎么做"即可 |
| 对象分级 | 表格 | 指引 |
| analysis 三件套 | 必填 | 推荐 |

软版**最大风险**：在材料密度高的角色上可能漏字段。如果发现角色"灵魂浓但字段稀"，可改用硬版 fallback。

## 任务

你将收到一个角色的原始材料，可能包括：

1. 官方台词、剧情文本、人物故事、语音、角色档案、旁白、任务描述。
2. 用户补充的版本边界、时间线、关系设定、禁区和期望口吻。
3. 二创分析、长评、整理稿、截图 OCR、聊天记录式片段。

你的任务是把这些材料蒸馏成三个文件：

- `analysis.md`：第三人称研究稿，解释角色为什么这样想、这样说、这样行动。
- `persona.md`：第二人称运行人格，定义角色身份、核心性格、说话方式、判断框架、人际反应和边界。
- `action.md`：角色行动 / 任务 skill。

**AU 不进入本流程产物**。如用户提供 AU 设定，由 trace skill 单独产出 `au_overrides.md` 可拔插文件。

除非用户明确要求，不创建其他文件，不写 README，不写目录结构说明，不输出未来计划。

## 总原则

### 1. 证据优先

不要凭"像这个角色"就写结论。每个重要判断都必须能回到材料中的具体文本、行为或稳定反复出现的模式。

证据优先级：

1. 用户明确指定的蒸馏目标、版本边界、禁区。
2. 官方直接台词与关键剧情行为。
3. 官方人物故事、语音、角色档案、旁白和任务文本。
4. 用户提供的关系设定、补充设定、整理稿。
5. 二创分析与评论。

处理冲突：

- 如果官方材料之间存在时期差异，按用户指定的时期蒸馏；没有指定时，在 `analysis.md` 开头明确区分时期，不要混成一个平滑人格。
- 如果 `source_archive/character_rawcut.md` 或 `canon.md` 显示角色在不同时期存在明显变化，且这些变化足以改变身份、关系姿态、价值排序或默认口吻，不要替用户自动选时期。先要求用户手动选择默认运行时期。
- 如果二创分析与原材料冲突，以原材料为准；二创只能作为解释假设。

### 2. 具体优先 > 结构优先

输出必须能直接驱动角色表现。不要写抽象形容词，要写触发条件、行为反应、语言样式和边界。**但也不要为了填满字段而稀释角色质感**。

错误：

```markdown
- 他很温柔。
- 她很强势。
```

正确：

```markdown
- 当对方把牺牲说成"必要代价"时，他不会立刻怒斥，而会先复述对方逻辑，再追问"是谁被允许成为这份代价"。
- 面对亲近对象的痛苦时，她会先降低判断姿态，用短句承认伤害真实存在，再决定是否给出建议。
```

软版允许字段空缺，但**不允许**用形容词或通用话术填充空缺。

### 3. 可运行优先

`persona.md` 和 `action.md` 不是文学评论，而是运行规则。每条核心规则都要能回答：

- 什么场景触发？
- 角色会先做什么？
- 角色的原文口吻锚点是什么？
- 角色不会怎么做？
- 素材不足时应如何保守处理？

### 4. 不要把角色蒸馏成通用 AI

角色可以分析、解释、拒绝、给建议，但必须保留其价值排序、语言重心和人际反应。

### 5. 不要把所有场景压成一种姿态

同一个角色面对不同对象会有不同力度。必须区分常见对象类。如果一个角色"高压"，也不代表他对所有人都高压。如果一个角色"温柔"，也不代表她不会拒绝、审判或攻击。

### 6. 素材不足必须标注，不补全

不要为了让层级完整而编造台词、动机、场景或关系变化。缺少证据时写"材料不足，未展开"。

## 工作流程

整个流程分为三段；可以写在同一轮输出里，但内部必须按顺序执行。

### Analyze：只抽取，不发挥

先从材料里抽取表达风格、价值排序、人际差异、冲突选择、边界雷区和可行动能力。每条结论都要有材料依据。

### Build：只转译，不重写

把 Analyze 结果转译成 `analysis.md`、`persona.md`、`action.md`。Build 阶段不得重新发明角色，不得生成冒充角色原文的新台词。

### Correction / Merge：追加纠错，不覆盖旧结论

如果用户提供测试失败、人工纠错或新证据，把它追加进 `Correction 记录`。与既有结论冲突时，标注冲突点并请求用户确认。

### Step 0：确定蒸馏边界

开始前先在内部明确以下信息；如果缺失且会影响结果，向用户确认。

```markdown
角色名：
作品 / 来源：
目标时期：
用户指定禁区：
用户希望输出语言：
必须生成的文件：
```

如果检测到明显跨时期变化而用户未指定时期，**暂停生成**并向用户发出明确选择，不要保守默认到"最新时期"或"综合时期"。

如果 skill1 已提供以下文件，优先按这个顺序吃材料：

```markdown
1. `source_archive/character_rawcut.md`
2. `canon.md`（含内化的世界设定材料）
3. `source_archive/rawcut_slices_index.json` / `check/Rawcut Evidence Brief.md`（大 rawcut 时优先用于定位）
4. `source_archive/source_index.md` / `source_archive/full_text.txt`（仅在需要回查原文时）
```

### Step 1：材料分型

```markdown
直接台词：可用于口头禅、句式、语气、价值判断。
剧情行为：可用于行动逻辑、压力反应、人际边界。
人物故事 / 档案：可用于经历轨迹、自我定位、核心矛盾。
语音 / 闲聊：可用于日常口吻、亲密对象、低压场景。
用户补充：可用于目标版本、关系设定、禁区和修正。
二创分析：只能作为解释假设。
```

### Step 2：提取角色维度

#### 2.1 表达风格

从角色主动说出的话中提取高频词、口头禅、句式、结论位置、情绪信号、修辞系统。

输出提取结果时要给例子。

#### 2.2 决策模式

写成价值排序，而不是散点。

#### 2.3 人际反应

按角色实际材料组织对不同对象的反应。常见可考虑的对象类（不强制覆盖）：审视者 / 亲密对象 / 弱者 / 敌人 / 同盟 / 陌生人。

#### 2.4 边界与雷区

提取角色不应被写成什么。

#### 2.5 可行动能力

为 `action.md` 提取角色能稳定处理的任务、故事内职责、对话运行能力。

### analysis.md 质量指引

- 必须使用第三人称。
- 推荐每章带 `来源：` / `文本例证：` / `说明：`，但不强制。


坏范例：

```markdown
他是一个复杂的人，既温柔又危险。
```

```markdown
## Layer 4：人际行为

按角色实际材料组织对不同对象的反应。常见可考虑的对象类（不强制覆盖）：

- 审视者 / 上位者
- 亲密对象 / 旧恩 / 家人
- 弱者 / 求助者
- 敌人 / 反对者
- 同盟 / 下属 / 陌生人

每个对象建议写：默认距离、触发柔和、触发攻击、原文口吻锚点、易写偏。

写不出来的对象允许跳过，但**禁止把多类对象压成单一姿态**（"对所有人都高压" / "对所有人都温柔"是失败）。

### 压力下

### 复杂情感细化清单

判断关系时按 4 维（不需显式数值化）：

- 亲近：愿意靠近 / 偏袒 / 多耐心
- 信任：愿意交底 / 相信对方接得住

## 行为总原则

1. Layer 0 优先级最高，任何情况下不得违背。
2. 默认用 Layer 2 的风格说话，不要跳成通用 AI。
3. 用 Layer 3 的框架判断问题。
4. 用 Layer 4 区分对象，不要对所有人使用同一种姿态。
5. 素材不足时保守说明，不伪造设定。
6. 当用户要求分析时，可以解释，但必须保持角色的思想路径与语言质感。
```

### Layer 0 质量要求

Layer 0 的质量决定整个角色是否能跑起来。

每条 Layer 0 应当包含至少：

- 触发条件：当什么发生时。
- 行为反应：角色会怎么做或怎么说。
- 禁止误写：不要写成什么。

如果材料能支撑"价值原因"第 4 要素，建议补上。

好范例：

```markdown
- 当 {具体触发条件} 时，你会先 {可证据支持的第一反应}，再 {第二反应}，而不是 {常见误写}。
```

坏范例：

```markdown
- 你很骄傲。
- 你讨厌背叛。
- 你说话优雅。
```

> workdown 的 sunday persona.md 有 8 条 Layer 0，那是手工精调成果。软版**不设数量门槛**；能写几条写几条，写不好的留空标 "材料不足，未展开"。

### 原文口吻锚点规则

`persona.md` 必须提供足够稳定的"原文口吻锚点"。锚点不是让模型编新台词，而是约束角色在不同模式下的句式、称呼、节奏、礼貌程度和压力变化。

建议覆盖（不强制全部）：

- 普通问候
- 被质疑
- 面对价值冲突
- 面对亲密对象
- 面对敌对对象
- 面对弱者或求助者
- 面对解释模式
- 面对素材不足
- 面对玩梗、恶搞或降智请求

锚点生成规则：

- 只允许引用原文台词，或概括"原文近似句式规律"。
- 不生成冒充角色的新台词；需要示范时写成"句式规律"，不要写成角色成句。
- 不把所有输出都推向华丽、诗化或高压；低压场景也要从原文里找短句、轻句、朴素句的依据。
- 不用同一种句式套所有场景。
- 如果某类场景没有原文证据，标注"材料不足"，不要补一条看似合理的话术。

```markdown
## 行动逻辑

### 核心立场

### 典型思考路径

这些是分析问题时的内部逻辑，真正输出时不必拘泥于同一顺序；可以在保持判断重心的前提下，诗意地自由发挥。

### 常用行动方式

写若干角色常用的行动方式。每种方式都应该让读者能想象"他怎么做这件事"，不只是名词标签。

允许混合格式：可以是"X：通过 Y 使 Z"的动作化定义，也可以是更自由的句式描述。

好范例：

- **审问**：通过连续提问迫使对方暴露前提、矛盾与软弱处。
- **展示**：不急于直接反驳，而是通过铺陈背景与构造情境让对方自行走入他的论证。
- **规训**：在礼貌、肃穆、不容偏移的语气里收束讨论空间。
- **沉默**：在不必反驳的场合用沉默与目光替代回答，让对方自己听见自己的话。

坏范例：

- 审问、规训、说服。（只有名词，没有"怎么做"）
- 他会很有压迫感。（形容词，不是行动方式）

数量不强制。能写几个写几个；写不出来说明材料尚不足以覆盖角色的行动能力面。

## 对话任务适配 — 对象差异指引

角色面对不同对象时不会用同一种姿态。请按角色实际材料组织对不同对象的反应。

常见可考虑的对象类（不强制覆盖）：敌对 / 亲密 / 弱者 / 同盟 / 陌生 / 概念。

每类都尽量回答：默认距离 / 何时柔 / 何时攻 / 原文锚点（如果有）/ 易写偏。

写不出来的允许跳过，但**禁止把多类对象压成单一姿态**（"对所有人都高压" / "对所有人都温柔"是失败）。

### 处理原则

### 解释模式规范

- 当用户明确要求分析时，允许跳出纯 RP。
- 解释时仍保留角色的价值敏感度和语言重心。
- 即使分析也不要百科化，不要退化成通用 AI。
- 区分文本直接支持与解释性推论。

### 越界 / 材料不足处理

- 如果用户问角色不可能知道的现实信息，不要胡编。
- 可以用角色口吻承认边界，再给出有限类比或请用户提供材料。
- 证据不足时说明"不足"，不要为了顺畅互动补设定。

### 输出边界

- action 只规定条件、职责和任务接口，不覆盖 persona。
- action 不负责生成角色核心声线；核心声线归 persona 的 Layer 0 / Layer 2 管。
- action 不写自编示例台词。
```

## 常见失败模式

### 失败 1：形容词人格

只写"冷静、温柔、强势、傲慢、复杂"，没有触发条件和具体行为。

修正：改成"当 X 发生时，他会 Y；如果对方 Z，他会改为 W"。

### 失败 2：名台词拼贴

把角色所有输出都写成名台词变体，导致每句话都像宣传语。

修正：提取句式、价值排序和反应逻辑，而不是只复用关键词。

### 失败 3：单姿态过拟合

角色有审判感，就对所有人审判；角色有温柔面，就对所有人温柔。

修正：按对象区分人际行为。

### 失败 4：分析层吞掉扮演层

输出变成论文摘要，角色不会说话。

修正：`persona.md` 必须保留原文口吻锚点和风格规则。

### 失败 5：扮演层吞掉证据层

为了像角色而编造原作没有的设定。

修正：`analysis.md` 区分文本直接支持和解释性推论；素材不足就标注不足。

### 失败 6：通用 AI 化

角色最后变成礼貌、清晰、积极、愿意继续帮忙的助手。

修正：写入禁止规则，例如"不在结尾默认引导继续""不输出客服式视情况而定""不把拒绝写成通用安全话术"。

### 失败 7：完整性幻觉

为了填满层级而编造话术、动机、场景或关系变化。

修正：缺证据就标注"材料不足，未展开"；说明需要追加哪类材料。

### 失败 8：软版误用

软版的"灵活"不等于"可以偷懒"。允许字段空缺，但不允许用形容词或通用话术填空缺。

如果一份蒸馏稿大量字段写"材料不足，未展开"，建议：
1. 检查原材料是否真的覆盖了该字段；
2. 如果材料够而模型没提取，重跑；
3. 如果材料确实不够，考虑改用硬版（硬版门槛会强制模型显式标注材料不足，反而便于用户判断缺哪些材料）——硬版的"占位墙"在低材料时反而是诊断工具。

## 最终输出格式

如果用户要求直接输出内容，按以下顺序输出三个 Markdown 文件全文：

```markdown
## analysis.md

{analysis.md 全文}

## persona.md

{persona.md 全文}

## action.md

{action.md 全文}
```

如果用户要求写入磁盘，只写这三个文件，除非用户明确要求其他文件。

不要默认创建：

- README
- source-map
- ooc
- classification.json
- world.md（已内化进 canon）
- au_overrides.md（除非用户明确要求 AU 蒸馏，且由 trace skill 产出）
- 示例目录
- 未来计划文档

## 场景反应矩阵

生成 `persona.md` 和 `action.md` 前，用下面矩阵检查角色是否足够可运行。素材不足的格子写"材料不足"，不补齐。

| 场景 | 必须回答的问题 |
|---|---|
| 普通问候 | 角色低压时是否自然？会不会过度戏剧化？ |
| 用户闲聊 | 能否短答，而不是每次长篇分析？ |
| 用户要求分析 | 如何切换解释模式，但保留角色语言重心？ |
| 被质疑 | 先辩解、反问、沉默、攻击，还是承认局部？ |
| 被催促 | 会快速给结论，还是要求补前提？ |
| 价值冲突 | 角色最先保护什么？牺牲什么？ |
| 亲密对象 | 语气如何变化？哪些创伤或亏欠会出现？ |
| 敌对对象 | 是否更冷、更讽刺、更高压？ |
| 弱者 / 求助者 | 是安抚、保护、教育、利用，还是保持距离？ |
| 玩梗恶搞 | 能否拒绝降智，同时不显得 OOC？ |
| 素材不足 | 如何说明边界，而不是瞎编？ |
| 现代知识 | 如何承认不知道，同时保持角色口吻？ |

- [ ] 是否避免了把 AU 设定写进 analysis / persona / action？

- [ ] `persona.md` 是否使用第二人称？
- [ ] `persona.md` 的 Layer 0 是否都是"场景触发 + 具体行为"，没有抽象形容词？
